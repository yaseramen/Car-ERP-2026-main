## أولاً: ما تم تنفيذه (منذ بداية المشروع حتى الآن)

### أساس المنصة

- **تطبيق ويب SaaS** (Next.js App Router، TypeScript، Tailwind، Turso/LibSQL).
- **مصادقة وجلسات** (NextAuth): تسجيل دخول، تسجيل شركات، استعادة كلمة مرور المالك.
- **أدوار:** Super Admin، مالك المركز (Tenant Owner)، موظف بصلاحيات شاشة تفصيلية.
- **شركات متعددة** مع عزل بيانات لكل شركة.
- **أنواع نشاط الشركة:** `sales_only`، `service_only`، `both`، **`supplier`** (مورّد — بدون ورشة/OBD في لوحة التصفح).

### المالية والمحافظ

- **محفظة شركة** وشحن/خصم من قبل Super Admin.
- **معاملات محفظة** مرتبطة بالخدمات الرقمية، OBD، المساعد، و**إعلانات السوق** (`marketplace_ad`).
- **خزائن** (بيع / ورشة)، وموزّعين مع خزائن فرعية عند التفعيل.
- **طرق دفع** متعددة على الفواتير.

### المخزون والمشتريات والكاشير

- **مخازن متعددة**، أصناف، حركات مخزون، باركود.
- **طباعة ملصقات باركود** (تحسين للطباعة الحرارية: PNG عالي الدقة، إطار الطباعة).
- **فواتير شراء وبيع وصيانة** مع تسلسل أرقام.
- **كاشير:** سلة، خصومات، ضريبة، مسودة محلية، دعم أوفلاين (طابور).
- **سعر الشراء** يظهر لمالك الشركة وSuper Admin فقط (بحث، السلة، مربعان تحت البحث قبل الإضافة)، مع تنبيهات هامش/خسارة.
- **مسودة فاتورة شراء** في التخزين المحلي (مثل الكاشير).
- **عرض أسعار قابل للطباعة** مع **خصومات حجمية متدرجة** (شرائح حسب إجمالي القائمة، حفظ في المتصفح).
- **إصلاح بحث الكاشير:** رفع حد جلب الأصناف من الـ API + **بحث خادمي** أثناء الكتابة حتى لا يختفي منتج موجود لأنه خارج أول دفعة محمّلة.

### الورشة وOBD

- **سير عمل الورشة:** استلام → فحص → صيانة → جاهزة → فاتورة.
- **OBD:** بحث، رفع تقرير، دفعات يدوية، تحليل بالوصف، بيانات حية؛ خصم من المحفظة حسب العملية.
- **طباعة تقارير OBD** بخلفية بيضاء (إصلاح طباعة الوضع الداكن).
- **واجهة OBD للتابلت:** تبويبات قابلة للتمرير، تحسين النماذج؛ استبدال صياغة «ذكاء اصطناعي» بـ **«أداة EFCT»** في النصوص والـ API.

### التسويق والواجهة العامة

- **صفحة هبوط** مع تحسينات للتابلت وأهداف لمس.
- **توجيه HTTP → HTTPS** عند تعريف `NEXT_PUBLIC_APP_URL` بصيغة https.

### السوق والمورّدين (Marketplace)

- **أعلام شركات:** `marketplace_enabled`، `ads_globally_disabled`؛ تسجيل مورّد جديد بسوق معطّل حتى مراجعة الإدارة.
- **جداول:** باقات إعلان، إعلانات (`marketplace_listings`)، شاشة صلاحيات `marketplace`.
- **صفحة عامة** `/market` (قطع غيار / ورشة) — عرض فقط، تواصل مع المورّد؛ دعم **الوضع الداكن** للنصوص والبطاقات.
- **سوبر أدمن:** نموذج **إعلان تجريبي** من `/admin/marketplace` (بدون خصم محفظة؛ `wallet_tx_id` فارغ) يظهر في السوق حتى لو كانت أعلام الشركة معطّلة؛ **المعلن يُختار تلقائياً** (أول شركة مورّد ثم مختلط) مع إمكانية تعديل يدوي اختياري؛ الـ API يقبل فقط `supplier` أو `both`؛ رفع صورة صغيرة كـ data URL أو رابط.
- **لوحة المورّد** `/admin/marketplace`: إنشاء إعلان بخصم محفظة + موافقة، تجديد، إلغاء، تجديد تلقائي اختياري.
- **سوبر أدمن:** نفس المسار لإدارة الباقات (تفعيل/تعطيل) وإيقاف إعلان طوارئ؛ إدارة الأعلام من المحافظ.
- **Cron** انتهاء الإعلانات وتجديد تلقائي عند الرصيد (`CRON_SECRET`؛ على Vercel Hobby **جدولة يومية** كحد أقصى).
- **تنبيه انتهاء قريب (48 ساعة)** شريط في لوحة التحكم للمورّد.

### قاعدة البيانات والنشر

- **مخطط + ترحيلات** في `database/`؛ **`npm run db:migrate`** ضمن `npm run build` على Vercel (فشل الترحيل يفشل النشر).
- **ترحيل مخطط `schema.sql` جملة جملة** لتفادي فشل Turso على دفعات ضخمة.

### اختبارات وصيانة

- إعداد **Vitest** لاختبارات وحدة (مثال: منطق خصم حجم عرض السعر).

## ثانياً: ما نريد تنفيذه لاحقاً (خطط تطوير)

### السوق والإعلانات

1. **تنبيهات قبل انتهاء الإعلان** عبر بريد و/أو واتساب و/أو إشعار داخل النظام (حالياً: شريط 48 ساعة فقط).
2. **رفع صور على تخزين المنصة** وحذف الملف عند انتهاء/إلغاء الإعلان (بدل URL فقط).
3. **Cron أكثر تكراراً من يومي** — ترقية Vercel Pro أو cron خارجي يستدعي `/api/cron/marketplace-expiry`.
4. **تحسين لوحة السوبر أدمن للسوق:** تعديل أسعار/مدة الباقات من الواجهة بسهولة؛ تقرير إيرادات إعلانات منفصل.
5. **مراجعة قانونية** لنصوص التنبيه والمسؤولية مع مستشار عند الحاجة.

### كاشير / مخزن

- متابعة إن تكررت شكاوى «منتج غير ظاهر»: موزّع المبيعات، `is_active`، فلتر القسم.

### بنية تقنية

- **Next.js 16 / proxy:** متابعة توصيات الانتقال من `middleware` حسب الوثائق الرسمية.
- **استقرار النشر:** عند تعطل webhook GitHub ↔ Vercel: إعادة ربط Git أو مراجعة Webhooks.

### عام

- أي ميزة جديدة يتفق عليها مالك المشروع تُضاف تحت هذا القسم مع تاريخ مختصر.

### طلبات شحن المحفظة بالإيصال (اعتماد يدوي — Super Admin)

> **قرارات مُثبَّتة:** حد أدنى للطلب **50 ج.م** (واجهة + API). **أكثر من طلب معلّق** لنفس الشركة مسموح. **الرفض** يتطلب **سبباً**. **السوبر أدمن** يعتمد **مبلغ الشحن الفعلي** (أقل أو أكثر من مبلغ الطلب) مع **تعليق**. **إيصال عبر Vercel Blob** كباقي المرفقات؛ **حتى 5 صور إيصال مُخزَّنة لكل شركة** للطلبات **المنفَّذة** (مقبول/مرفوض) مع حذف **الأقدم** عند التجاوز — **بدون حذف تلقائي لإيصال طلب معلّق**. إعلام الشركة عند **اكتمال الشحن** (قبول): إشعار داخل اللوحة + إشعار متصفح عند التفعيل (ومسار بريد لاحقاً إن وُجد).

#### المرحلة 1 — قاعدة البيانات والنماذج

1. ترحيل جديد: جدول `wallet_topup_requests` (أو اسم مقارب) بحقول مثل: `id`, `company_id`, `requested_by` (user), `requested_amount`, `receipt_blob_url` (أو مفتاح Blob), `status` (`pending` | `approved` | `rejected`), `approved_amount` (nullable), `admin_comment` (nullable), `reject_reason` (nullable), `wallet_transaction_id` (nullable — ربط بـ `wallet_transactions` عند القبول), `processed_by`, `processed_at`, `created_at`.
2. فهارس: `company_id` + `status`، و`created_at` لقائمة الأدمن ولسياسة التنظيف.
3. تحديث `database/schema.sql` ليعكس الجدول نفسه (للمطابقة المرجعية).

#### المرحلة 2 — API الشركة (طلب شحن)

1. `POST` إنشاء طلب: مصادقة **مالك/مستخدم بصلاحية شاشة المحفظة**، التحقق من `requested_amount >= 50`، رفع الصورة إلى Blob (نفس أنماط `marketplace` / الملاحظات إن وُجدت)، تسجيل الصف في الحالة `pending`.
2. `GET` قائمة طلبات الشركة (اختياري للواجهة): معلّق + منتهٍ مع التفاصيل الظاهرة للعميل.
3. منع الحذف التلقائي لBlob لأي صف `pending` (انظر المرحلة 5).

#### المرحلة 3 — API السوبر أدمن

1. `GET` قائمة الطلبات المعلّقة (وفلترة اختيارية بالشركة).
2. `POST` أو `PATCH` **قبول**: `approved_amount > 0` (مع تحقق حد أدنى 50 إن رُغب توحيداً مع سياسة الشحن)، `admin_comment` اختياري؛ تنفيذ نفس منطق الشحن الحالي (`wallet_transactions` + تحديث `company_wallets`) مع `reference_type`/`reference_id` يشيران لطلب الإيصال؛ تحديث الصف إلى `approved`؛ تسجيل `performed_by`.
3. **رفض**: `reject_reason` إلزامي؛ تحديث الحالة فقط (بدون رصيد).
4. تسجيل في **سجل التدقيق** إن كان المشروع يستخدمه لعمليات مالية حساسة.

#### المرحلة 4 — إشعارات الشركة والأدمن

1. **بعد القبول:** إنشاء سجل إشعار يقرأُه مالكو/مستخدمو الشركة (جدول إشعارات أو توسيع آلية موجودة) بنص يوضح أن **الشحن تم** والمبلغ المعتمد؛ تكرار نمط **إشعار المتصفح** الموجود في `NotificationsProvider` عند وجود عداد/نقطة نهاية جديدة.
2. **طلب جديد معلّق:** إشعار/عداد للسوبر أدمن (قائمة + اختياري متصفح) مشابه لملخص الملاحظات.
3. **بعد الرفض:** إشعار للشركة مع **سبب الرفض** (بدون «شحن تم»).

#### المرحلة 5 — Blob: حد 5 صور لكل شركة (منفَّذ فقط)

1. عند **القبول أو الرفض** (بعد ثبات الحالة): عدّ صفوف الطلبات المنفَّذة لدى الشركة التي لديها `receipt_blob_url`؛ إن تجاوز **5**، احذف من التخزين **أقدم** URL مرتبط بطلب **ليس** `pending` (أي `approved`/`rejected`)، ثم امسح الحقل أو علّم أن الملف أُزيل.
2. **لا** تشغيل هذه الدورة على طلبات `pending`.
3. حذف يدوي لطلب معلّق (إن أُضيف لاحقاً) يبقى **فقط** بقرار أدمن إن وُجدت ميزة إلغاء.

#### المرحلة 6 — واجهات المستخدم

1. **لوحة المحفظة (الشركة):** نموذج «طلب شحن» (مبلغ ≥ 50، رفع إيصال)، قائمة بحالة الطلبات، عرض سبب الرفض ورسالة نجاح الشحن.
2. **لوحة السوبر أدمن:** صفحة أو قسم «طلبات شحن المحفظة» — معاينة الإيصال، حقول المبلغ المعتمد والتعليق، أزرار قبول/رفض (الرفض بدون إرسال بدون سبب).
3. ربط من قائمة المحافظ الحالية إن لزم.

#### المرحلة 7 — اختبارات وجودة

1. اختبارات وحدة: التحقق من الحد الأدنى، انتقال الحالات، عدم حذف blob لـ `pending`.
2. اختبار يدوي: طلب متعدد لنفس الشركة، قبول بمبلغ مختلف، رفض بسبب، إشعار الشركة، دورة الـ 5 صور.
